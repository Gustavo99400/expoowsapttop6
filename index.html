02 ¿QUÉ ES OWASP Y A06?

>_

OWASP TOP 10

OWASP (Open Web Application Security Project) es una fundación global sin fines de lucro enfocada en mejorar la seguridad del software. Su documento de referencia, el Top 10, representa el consenso general sobre los riesgos de seguridad más críticos en las aplicaciones web.

A06:2025 – INSECURE DESIGN

Incorporado para representar los riesgos relacionados con fallas en el diseño y la arquitectura. Se centra en la necesidad de incorporar el modelado de amenazas, patrones de diseño seguros y arquitecturas de referencia robustas.

03 IMPORTANCIA DEL PROBLEMA // ¿POR QUÉ ES CRÍTICO?

DIFÍCIL DETECCIÓN

No se puede probar la seguridad en un diseño inexistente o deficiente. Las herramientas automáticas de análisis de código (SAST/DAST) rara vez detectan errores conceptuales o de lógica de negocio, haciendo que requiera revisiones de arquitectura humanas.

PLANIFICACIÓN DESDE EL INICIO

La seguridad no es una característica que se añade al final del desarrollo. Debe planificarse desde la especificación de requisitos. Corregir un error de diseño en producción puede costar hasta 100 veces más que resolverlo en la etapa de diseño.

IMPACTO EN TODO EL SISTEMA

Las fallas de diseño comprometen la base del software. Si la arquitectura no define límites de confianza y control claros, los parches a nivel de código solo ocultarán temporalmente una vulnerabilidad estructural mucho mayor.

"Un diseño seguro puede resistir errores de implementación inevitables, pero un diseño inseguro no se puede corregir con una codificación perfecta."

04 ¿QUÉ ES EL DISEÑO INSEGURO?

Es una vulnerabilidad que surge cuando una aplicación o sistema es concebido, diseñado o estructurado sin los controles de seguridad necesarios para proteger sus recursos y flujos de negocio.

No se trata de líneas de código mal escritas (como no sanitizar una entrada); se trata de procesos de negocio inherentemente vulnerables, falta de segregación de privilegios, o ausencia de límites técnicos en la lógica del sistema.

Ciclo de Diseño: Seguro vs Inseguro

Proceso Inseguro (Tradicional)
1. Requisitos de Negocio ➔
2. Diseño Rápido (Seguridad Omitida) ➔
3. Codificación sin Guías ➔
4. Despliegue Vulnerable
Proceso Seguro (SSDLC)
1. Requisitos con Seguridad ➔
2. Modelado de Amenazas ➔
3. Codificación Segura ➔
4. Despliegue Validado

05 CAUSAS PRINCIPALES // ¿CÓMO SE ORIGINA?

01

Falta de Requisitos de Seguridad

No se especifican las necesidades de protección al iniciar el proyecto.

Haga clic para ver más

Falta de Requisitos

El desarrollo comienza únicamente basándose en la funcionalidad deseada por el cliente. No se definen políticas de contraseñas, encriptación, retención de datos o niveles de auditoría indispensables.

Volver
02

Ausencia de Modelado de Amenazas

No se analizan los vectores de ataque antes del desarrollo.

Haga clic para ver más

Ausencia de Modelado

No se realiza una sesión de STRIDE u otra metodología para descubrir quién podría atacar el sistema y cómo. Como consecuencia, no se diseñan contramedidas para mitigar esos riesgos potenciales.

Volver
03

Funcionalidad sobre Seguridad

Se prioriza la rapidez de entrega del software sobre los controles.

Haga clic para ver más

Funcionalidad Primero

La presión por lanzar características al mercado ("Time to Market") desplaza la validación de seguridad. Se desactivan protecciones para evitar retrasos en el desarrollo o fricción con el usuario final.

Volver
04

Controles de Acceso Deficientes

Estructuras de autorización mal planteadas en la raíz del sistema.

Haga clic para ver más

Diseño de Autorización

La lógica de control de accesos se construye de forma ad-hoc en lugar de usar un sistema centralizado y robusto. Esto facilita la elusión de restricciones mediante manipulación de parámetros directos.

Volver
05

Falta de Validación en Flujos Críticos

Se asume que los usuarios seguirán el flujo de la aplicación.

Haga clic para ver más

Flujos Críticos

Se confía en el estado del lado del cliente. Un usuario malicioso puede saltarse pasos intermedios (como el pago o la verificación de identidad) y comunicarse directamente con el endpoint final del servidor.

Volver
06

Arquitectura Insegura

Exposición indebida de APIs o componentes sin aislamiento.

Haga clic para ver más

Arquitectura Defectuosa

Los microservicios internos se comunican sin autenticación, o los datos altamente confidenciales se almacenan en zonas accesibles por componentes de presentación sin cifrado ni límites claros.

Volver

07 CONSECUENCIAS E IMPACTO

Las fallas de diseño tienen consecuencias severas y duraderas.

A diferencia de un bug simple que se parcha con una línea de código, corregir un diseño inseguro suele requerir la reescritura completa del sistema o cambios profundos en las bases de datos y APIs.

  • Fraude Financiero Directo: Los abusos de lógica permiten transacciones sin fondos o desvíos de dinero.
  • Daño Reputacional Crítico: Pérdida de la confianza de clientes y socios comerciales ante fallos evitables.
  • Sanciones Legales: Multas severas por violaciones de normativas de protección de datos (PCI-DSS, GDPR).

08 CÓMO IDENTIFICAR // AUDITORÍA Y ANÁLISIS

Tácticas de Evaluación de Diseño

Selecciona una táctica para auditar la arquitectura:

Modelado de Amenazas

OBJETIVO

Identificar posibles ataques y fallas arquitectónicas antes de escribir código.

¿QUÉ SE EVALÚA?

Activos del sistema, límites de confianza, flujos de datos y amenazas potenciales (STRIDE).

RESULTADO ESPERADO

Conocer los riesgos potenciales del software y diseñar mitigaciones efectivas antes del desarrollo.

09 MEDIDAS DE PREVENCIÓN // BUENAS PRÁCTICAS

1. Seguridad por Diseño (SBD)

Incorporar expertos en seguridad y requisitos de seguridad desde la fase inicial de definición del proyecto.

2. Mínimo Privilegio

Principio

Diseñar los accesos para que las cuentas de usuario y servicios internos solo tengan los permisos estrictamente necesarios.

3. Modelado de Amenazas Continuo

Ejecutar análisis de amenazas sistemáticos para todos los flujos de negocio críticos antes de su desarrollo.

4. Defensa en Profundidad

Implementar múltiples capas de control alrededor de los activos críticos en lugar de confiar en una sola barrera perimetral.

5. Validar en Servidor

Urgente

No delegar validaciones ni control de estado crítico al navegador o cliente. Toda regla de negocio se valida en el servidor.

6. Auditorías de Arquitectura

Realizar revisiones periódicas de los patrones de diseño del software para garantizar que se adaptan a las nuevas amenazas.
